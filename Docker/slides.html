01

¿Por qué Docker?

El problema antes de la solución

El Problema Clásico

"En mi máquina funciona"

...dijo todo desarrollador, al menos una vez

¿Por qué pasa esto?

  • Distintos sistemas operativos (Windows, macOS, Linux)
  • Distintas versiones de librerías instaladas
  • Distintas variables de entorno y configuraciones
  • Distintas dependencias del sistema

Configurar un entorno de desarrollo puede llevar horas o días

La Promesa de Docker

  • Empaquetá tu app con todas sus dependencias
  • Ejecutala en cualquier lugar — tu laptop, el servidor, la nube
  • Mismo entorno para todo el equipo

¿Quién usa Docker?

Desarrollo

  • Entornos de desarrollo local
  • Onboarding de nuevos miembros
  • Testing y CI/CD

Producción

  • Despliegues en la nube (AWS, GCP, Azure)
  • Arquitectura de microservicios
  • Aplicaciones web escalables
02

Servicios y Puertos

Cómo los procesos se comunican con el exterior

¿Qué es un Servicio?

Un proceso que corre continuamente y escucha peticiones

  • Un servidor web (Nginx, Apache)
  • Una base de datos (PostgreSQL, MySQL)
  • Tu aplicación (Node.js, Flask, Spring)

Un servicio siempre está esperando que alguien le pida algo

¿Qué es un Puerto?

El "número de puerta" donde escucha un servicio

Tu Servidor

:80 HTTP :443 HTTPS
:5432 Postgres :3306 MySQL
:6379 Redis :3000 Tu App

Una sola máquina puede correr muchos servicios — cada uno en un puerto diferente

Mapeo de Puertos en Docker

Docker permite mapear un puerto del host a un puerto del contenedor


docker run -p 8080:80 nginx

Host

:8080
→

Contenedor

:80

Visitá localhost:8080 en el navegador → llega a Nginx dentro del contenedor en el puerto 80

03

Virtualización

Dividir una máquina en muchas

¿Qué significa "virtualizar"?

Crear una versión por software de algo que normalmente es físico

Analogía: Un servidor físico es como una casa

La virtualización lo divide en departamentos — cada uno independiente

Esto no es nuevo — IBM lo hacía en los años 60 con mainframes

¿Por qué aislar entornos?

  • Conflictos de dependencias
    La App A necesita Python 3.8, la App B necesita Python 3.12 — chocan
  • Seguridad
    Si una app se ve comprometida, no puede afectar a las demás
  • Reproducibilidad
    El mismo entorno en todos lados: desarrollo, testing, producción
  • Testing
    Levantar, probar, destruir — sin ensuciar tu sistema

Asignación de Recursos

Un servidor físico suele correr al 10-15% de utilización

Sin virtualización:
~12%

La virtualización permite empaquetar múltiples cargas de trabajo en el mismo hardware

Con virtualización:
~78%

Por esto el cloud computing es económicamente viable — vender porciones en vez de máquinas enteras

¿Cuándo usar cada uno?

VMs

  • Necesitás un SO diferente (ej: Windows en Linux)
  • Se requiere aislamiento fuerte de seguridad
  • Aplicaciones legacy

Contenedores

  • Microservicios
  • Pipelines de CI/CD
  • Entornos de desarrollo
  • Apps cloud-native

No son excluyentes — en producción, los contenedores suelen correr dentro de VMs

05

¿Qué es un Contenedor?

Aislado, liviano, efímero

Un Contenedor es...

  • Un proceso aislado corriendo en tu máquina
  • No es una VM liviana — es simplemente un proceso con límites
  • Tiene su propio:
    • Sistema de archivos
    • Interfaz de red
    • Espacio de procesos

Efímeros por Naturaleza

Los contenedores están pensados para ser destruidos y recreados fácilmente

No guardan estado valioso — tratalos como descartables

Ganado, no mascotas

No arreglás un contenedor roto — lo reemplazás

06

¿Qué es una Imagen?

La plantilla de los contenedores

Imagen vs Contenedor

Imagen

  • Una plantilla
  • Solo lectura
  • Puede crear muchos contenedores

Contenedor

  • Una instancia en ejecución
  • Capa de lectura-escritura encima
  • Creado a partir de una imagen

Docker Hub

Un registro público de imágenes preconstruidas


# Descargar una imagen oficial
docker pull nginx:latest
docker pull python:3.12
docker pull postgres:16
docker pull node:20-alpine

imagen:tag — el tag especifica la versión

Las variantes alpine son imágenes mínimas (~5MB de base)

07

Docker CLI

Manos a la terminal

Tu Primer Contenedor


# Descargar y ejecutar Nginx
docker run -d -p 8080:80 --name web nginx

-d → correr en segundo plano (detached)

-p 8080:80 → mapear puerto 8080 del host al 80 del contenedor

--name web → darle un nombre amigable

nginx → la imagen a usar

Abrí http://localhost:8080 en el navegador

Inspeccionando Contenedores


# Listar contenedores corriendo
docker ps

# Listar TODOS los contenedores (incluidos los detenidos)
docker ps -a

# Ver logs del contenedor
docker logs web

# Seguir logs en tiempo real
docker logs -f web

Interactuando con Contenedores


# Abrir una shell dentro del contenedor
docker exec -it web bash

# Ejecutar un comando puntual adentro
docker exec web cat /etc/nginx/nginx.conf

-it = interactivo + TTY → te da una terminal adentro

Comandos de Ciclo de Vida


# Detener un contenedor corriendo
docker stop web

# Iniciar un contenedor detenido
docker start web

# Eliminar un contenedor detenido
docker rm web

# Forzar eliminación de un contenedor corriendo
docker rm -f web

Comandos de Imágenes


# Descargar una imagen
docker pull python:3.12

# Listar imágenes locales
docker images

# Eliminar una imagen
docker rmi python:3.12

# Construir una imagen desde un Dockerfile
docker build -t myapp:1.0 .

08

Volúmenes

Datos persistentes en un mundo efímero

El Problema

El contenedor muere → los datos mueren

¿Qué pasa con tu base de datos? ¿Tus archivos subidos? ¿Tus logs?

Necesitás una forma de persistir datos fuera del contenedor

La Solución: Volúmenes

Contenedor

/var/lib/postgresql/data

↔

Volumen

pgdata (en disco del host)

Los datos viven en el host — sobreviven a la eliminación del contenedor

Dos Tipos

Volúmenes con Nombre

Gestionados por Docker


docker run -d \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16

Bind Mounts

Mapean un directorio del host directamente


docker run -d \
  -v ./mi-codigo:/app \
  python:3.12

Los bind mounts son ideales para desarrollo — editás código en tu host, los cambios se ven en el contenedor

09

Networking

Cómo se comunican los contenedores

Red Bridge por Defecto

Docker crea una red aislada para los contenedores

Máquina Host
docker0 bridge
web
172.17.0.2:80
api
172.17.0.3:3000
db
172.17.0.4:5432

Publicación de Puertos

Los contenedores están aislados por defecto — sin acceso externo

Usá -p para publicar un puerto hacia el host 


# Mapeo explícito
docker run -p 8080:80 nginx

# Mapear a un puerto aleatorio del host
docker run -P nginx

# Vincular a una interfaz específica
docker run -p 127.0.0.1:8080:80 nginx

Descubrimiento de Servicios en Compose

En Docker Compose, los contenedores se alcanzan por nombre de servicio 


# Dentro del contenedor "api", conectarse a la base de datos:
connection = psycopg2.connect(
    host="db",        # <-- solo el nombre del servicio!
    port=5432,
    database="myapp"
)

No hacen falta direcciones IP — Docker resuelve el DNS automáticamente

Drivers de Red

Driver Caso de Uso
bridge Por defecto. Contenedores en el mismo host se comunican por un puente virtual
host El contenedor comparte la red del host directamente (sin aislamiento)
none Sin red (totalmente aislado)

Para esta materia, bridge es todo lo que necesitan

10

Docker Compose

Aplicaciones multi-contenedor de forma simple

El Problema

Una app real tiene múltiples servicios


# Levantar todo manualmente...
docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres:16
docker run -d --name redis redis:7
docker run -d --name api -p 3000:3000 --link db --link redis myapp
docker run -d --name web -p 8080:80 --link api nginx

Tedioso. Propenso a errores. Difícil de reproducir.

La Solución: Compose

Definir todo en un solo docker-compose.yml


services:

  db:
    image: postgres:16
    environment:
      POSTGRES_DB: myapp
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data

  api:
    build: ./api
    ports:
      - "3000:3000"
    depends_on:
      - db

  web:
    image: nginx:alpine
    ports:
      - "8080:80"

volumes:
  pgdata:

Comandos de Compose


# Levantar todos los servicios (construir si es necesario)
docker compose up -d

# Ver servicios corriendo
docker compose ps

# Ver logs de todos los servicios
docker compose logs -f

# Detener y eliminar todo
docker compose down

# Detener, eliminar Y borrar volúmenes
docker compose down -v

¿Por qué Compose?

  • Un archivo describe todo tu stack
  • Un comando para levantar todo
  • Networking entre servicios es automático
  • Reproducible — commitealo a git, compartilo con el equipo

¿Nuevo integrante del equipo? git clone + docker compose up = listo

11

Dockerfile

Construyendo tus propias imágenes

¿Qué es un Dockerfile?

Un archivo de texto con instrucciones para construir una imagen

Como una receta — cada línea es un paso


# Todo Dockerfile arranca desde una imagen base
FROM python:3.12-slim

No arrancás de cero — te parás sobre los hombros de imágenes existentes

Instrucciones Clave

Instrucción Qué hace
FROMImagen base desde la que se parte
WORKDIREstablece el directorio de trabajo dentro de la imagen
COPYCopia archivos de tu máquina a la imagen
RUNEjecuta un comando durante la construcción (instalar paquetes, etc.)
EXPOSEDocumenta qué puerto usa la app
CMDComando por defecto al iniciar el contenedor

Ejemplo Práctico

Containerizando una aplicación Flask

app.py


from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello():
    return "Hola desde Docker!"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=5000)

requirements.txt


flask==3.0.0

Ejemplo Práctico

Dockerfile


FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

EXPOSE 5000

CMD ["python", "app.py"]

Construir y Ejecutar


# Construir la imagen
docker build -t mi-flask-app .
          

# Ejecutarla
docker run -d -p 5000:5000 --name flask mi-flask-app
          

# Probarla
curl http://localhost:5000
# Hola desde Docker!

De código a contenedor corriendo en 3 comandos

Resumen

  1. Docker resuelve el problema de "en mi máquina funciona"
  2. Los servicios escuchan en puertos — Docker los mapea entre host y contenedor
  3. La virtualización permite compartir hardware de forma eficiente
  4. Los contenedores son livianos, rápidos y efímeros (a diferencia de las VMs)
  5. Las imágenes son las plantillas — inmutables
  6. La CLI te da control total: run, stop, logs, exec
  7. Los volúmenes persisten datos más allá del ciclo de vida del contenedor
  8. El networking permite que los contenedores se comuniquen entre sí
  9. Compose orquesta apps multi-contenedor con un solo archivo
  10. Los Dockerfiles permiten construir tus propias imágenes
